Codigo Limpio

Es una manera de escribir un codigo auto documentado, para que otro humano lo entienda con solo leerlo.


Reglas:


1 -> Los nombres son importantes

Los nombres de variables/método siempre deben de responder a estas preguntas: (Para que existe, que hace, como se usa)


2 -> No repetir

No se puede tener código duplicado en un programa. hay que crear métodos que hagan las cosas indicadas.


3 -> las funciones deben de hacer una sola cosa

Si una función está muy grande hay que partirla en funciones más pequeñas.


4 -> Comentar solo lo necesario

Se supone que el codigo se autodocumenta. los comentarios solo deben ser de advertencia o aclaratorios.

En ves de escribir un comentario es mejor que el codigo explique lo que hace.

NOTA: Si el codigo cambia el comentario queda obsoleto y por tanto miente.


5 -> Dejar el codigo más limpio de lo que estaba

Si vas a tocar un pedacito de un codigo hay que mejorar su código a su alrededor.


6 -> Escribe Try/Cacth/Finally

En caso de error hay que dejar el programa preparado para que continue.


7 -> No devuelvas null

Muchos lenguajes no gestionan NULL entonces es muy peligroso.


8 -> Orden de declaración de variables

Primero: Constantes Publicas.

Segundo: Constantes Privadas.

Tercero: variables Publicas.

Cuarto: variables Privadas.

Quinto: Funciones Publicas.

Sexto: Funciones Privadas.


9 -> Utilizar copias de objetos para trabajar concurrencia

Los objetos deben ser inmutables, por ello se necesita crear copias de cada objeto.
